## 生草系统爬虫（可配置版 `kusa_auto_config.py`）

`kusa_auto_config.py` 是一个使用 Playwright 的自动化脚本，用于在浏览器中自动完成「生草」流程：登录 → 进入生草页 → 恢复承载力 → 选择草种 → 点击「过载生草」或「开始生草」→ 监控完成 → 写入信号文件 `kusa_done.json`。

本 README 只介绍 **`kusa_auto_config.py` 的用法**。

### 环境准备

- **Python**: 3.8+
- **依赖安装**：

```bash
pip install -r requirements.txt
playwright install chromium
```

### 基本用法

- **单次执行（默认模式）**：

```bash
python kusa_auto_config.py
```

含义：使用默认草种「巨草」，点击「过载生草」，等待直到检测到页面进入「正在生长中 / 生完 / 完成 / 巨草 / 半灵草」等状态后，写入 `kusa_done.json` 并退出。

### 常用参数（全部针对 `kusa_auto_config.py`）

- **草种 / 触发方式**：

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `--kusa-type` | 巨草 | 草种，例如 `巨草`、`草`、`半灵草` 等；会强制改写 `plantKusa` 的草种实参（单轮或未使用 `--cycle-kusa` 时生效） |
| `--trigger` | 过载生草 | 点击的按钮：`过载生草` 或 `开始生草` |
| `--cycle-kusa` | — | **循环模式**下轮流使用的草种，逗号分隔，如 `不灵草,灵灵草`；指定后每轮按顺序切换，忽略 `--kusa-type`。当精确为 `不灵草,灵灵草` 时会进入「特殊循环模式」，详见后文。 |

- **循环相关（`--loop` 模式）**：

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `--wait-min` | 5 | 每轮完成后，先等待的分钟数（支持小数，例如 `0.5` 表示 30 秒，默认 5） |
| `--poll-min` | 1 | 在等待结束后，每隔多少分钟检查一次按钮是否重新出现（支持小数，例如 `0.5` 表示 30 秒） |
| `--retry-min` | 1 | 本轮未在限定时间内检测到完成时，多少分钟后在同一浏览器中重试整轮流程（支持小数） |
| `--max-poll` | 10 | 轮询按钮的最多次数，超过仍未出现则认为本轮失败，重新跑一轮 |
| `--buling-fast` | 关闭 | 仅在「特殊循环模式 `不灵草,灵灵草`」下生效：为不灵草启用实验性催生装置，生不灵草时跳过 `wait-min` 的等待，直接开始轮询检查触发按钮是否刷新（理论上 50% 概率在 1 分钟内收获）。 |
| `--skip-lingxing-check` | 关闭 | 仅在「特殊循环模式 `不灵草,灵灵草`」下生效：关闭每轮开头的灵性标签检测，使特殊模式退化为按 `不灵草,灵灵草` 顺序轮换草种。 |
| `--yield-threshold` | 关闭 | 预知草精下限：当「草之精华」低于该值时除草重试；与 `--yield-grass-min` 同时使用时优先级为 草种 > 草精数 > 生草数，详见「草种筛选与预知草精」。 |
| `--yield-max-retry` | 3 | 预知草精模式下除草重生的最大尝试次数；**仅保护草种**模式下也用作「刷保护草种」的最大次数（为 0 时按 10 次计）。 |
| `--yield-protect-kusa` | 关闭 | 保护草种列表，逗号分隔，如 `灵草,半灵草`。实际生长草种（「某某草正在生长中」中的「某某草」）在列表中则直接保留。**可不与 `--yield-threshold` 同时使用**：仅指定本参数时，会循环「生草 → 检查草种 → 不在列表则除草重试」直到出现保护草种或达到最大次数；未找到除草按钮时会在当前页面重试寻找（最多 3 次，每次间隔 1 秒），找到并点击后再继续生草，连续 3 次未找到则保留当前草。 |
| `--yield-lianhao-min` | 3 | 预知草精模式下，草数字「连续相同数字」达到该位数时视为连号并忽略阈值；设为 **0** 则不考虑连号，始终按草精阈值筛选。 |
| `--yield-grass-min` | 关闭 | 预知生草数下限：预知产量中「草 N」低于 N 时除草重试；支持 **k**（千）、**m**（百万），如 `100k`、`1.5m`；**筛选优先级**为 草种 > 草精数 > 生草数 > 沼气倍率。 |
| `--yield-biogas-min` | 关闭 | 沼气倍率下限：页面中「沼气×1.n」格式（通常在灵性附近），仅当倍率**大于**该值且其他条件满足时才保留，否则除草重试；如 `1.5` 表示需大于 1.5。 |

- **登录与页面**：

| 参数 / 环境变量 | 默认值 / 说明 |
|----------------|---------------|
| `--qq` / `KUSA_QQ` | 登录用 QQ 号；优先级：命令行 `--qq` > 环境变量 `KUSA_QQ` > 代码中的 `LOGIN_QQ`（默认为 `530958461`） |
| `--url` / `KUSA_GAME_URL` | 游戏页面地址；优先级：命令行 `--url` > 环境变量 `KUSA_GAME_URL` > 内置默认 `http://110.41.149.62/kusa` |

- **浏览器显示**：

| 方式 | 说明 |
|------|------|
| 默认 | 无头模式（不弹出浏览器窗口） |
| `--no-headless` | 显示浏览器窗口，方便调试 |
| 环境变量 `HEADLESS=true` | 也可以通过设置该变量强制开启无头模式 |

### 运行示例

- **默认：巨草 + 过载生草，单次执行**：

```bash
python kusa_auto_config.py
```

- **指定草种为「草」，点击「开始生草」**：

```bash
python kusa_auto_config.py --kusa-type 草 --trigger 开始生草
```

- **循环模式：先等 5 分钟，再每 1 分钟检查按钮（最多 10 次）**：

```bash
python kusa_auto_config.py --loop
```

- **循环模式：先等 3 分钟，再每 0.5 分钟检查按钮，且使用「开始生草」**：

```bash
python kusa_auto_config.py --loop --wait-min 3 --poll-min 0.5 --trigger 开始生草
```

- **循环模式：不灵草与灵灵草轮流（每轮自动切换草种）**：

```bash
python kusa_auto_config.py --loop --cycle-kusa 不灵草,灵灵草
```

- **指定 QQ 与 URL（命令行方式）**：

```bash
python kusa_auto_config.py --qq 123456789 --url http://example.com/kusa
```

- **通过环境变量指定 QQ 与 URL**：

```bash
export KUSA_QQ=123456789
export KUSA_GAME_URL=http://example.com/kusa
python kusa_auto_config.py
```

- **仅保护草种模式（循环直到出现保护草种，不设阈值）**：

```bash
python kusa_auto_config.py --loop --yield-protect-kusa 灵草,半灵草 --yield-max-retry 10 --qq 你的QQ
```

- **预知草精 + 保护草种（阈值 3，保护灵草/半灵草，连号最少 4 位才保留）**：

```bash
python kusa_auto_config.py --loop --yield-threshold 3 --yield-protect-kusa 灵草,半灵草 --yield-lianhao-min 4 --qq 你的QQ
```

- **沼气倍率筛选（仅保留倍率大于 1.5 的草，可与草精/生草数等组合）**：

```bash
python kusa_auto_config.py --loop --yield-biogas-min 1.5 --yield-max-retry 5 --qq 你的QQ
```

### 信号文件 `kusa_done.json`

每当脚本认为本轮「生草」完成时，会：

1. 在控制台打印完成日志；
2. 在脚本所在目录写入/更新 `kusa_done.json`，格式类似：

```json
{
  "done": true,
  "at": "2025-02-27T12:00:00",
  "message": "过载生草已完成",
  "detected_by": "text=完成"
}
```

字段含义：

- **`done`**: 是否完成（恒为 `true`）；
- **`at`**: 服务器当前时间的 ISO 字符串；
- **`message`**: 本轮的提示信息（会根据触发方式有所不同）；
- **`detected_by`**: 是通过哪个页面元素/文本判断为完成的。

你可以：

- **直接查看文件**：生完草时 `kusa_done.json` 会被创建或覆盖；
- **编写监听脚本**：定时读取 `kusa_done.json` 或监控其变化，用于弹窗提醒、系统通知等。

### 不灵草 / 灵灵草轮流（`--cycle-kusa`）与特殊循环模式

在 `--loop` 下使用 `--cycle-kusa` 时有两种情况：

- **普通轮换模式**：例如 `--cycle-kusa 不灵草,半灵草,巨草`  
  - 每轮按给定顺序轮流使用草种：第 1 轮用列表第 1 个，第 2 轮第 2 个，…，到末尾后重新从第 1 个开始。  
  - 每轮开始前会打印当前草种（如 `[本轮草种] 不灵草`）。

- **特殊循环模式（精确为 `--cycle-kusa 不灵草,灵灵草`）**：
  - 默认行为（未加 `--skip-lingxing-check`）：
    - 每一轮开始时，脚本会先「登录 → 进入生草页 → 检查是否存在『灵性』标签」：
      - 若检测到灵性标签，则本轮草种强制为 **灵灵草**；
      - 若未检测到灵性标签，则本轮草种强制为 **不灵草**。
    - 控制台会打印类似：
      - `[special cycle] 本轮灵性标签检测结果: 有灵性 / 无灵性`
      - `[本轮草种] 灵灵草（基于当前灵性标签）` 或 `[本轮草种] 不灵草（基于当前灵性标签）`
    - **每一轮都是重新检测灵性再决定草种**，与第几轮、上一轮是什么草种无关。
  - 若同时指定 `--buling-fast`，则在本轮草种是「不灵草」并完成生草后：
    - 跳过 `wait-min` 的固定等待时间；
    - 直接进入「轮询触发按钮是否刷新」阶段，相当于为不灵草启用实验性催生装置（游戏内设定为：生不灵草时，50% 概率在 1 分钟内收获）。
  - 若指定 `--skip-lingxing-check`，则关闭上述灵性检测逻辑：
    - 特殊模式退化为「普通轮换模式」：按照 `不灵草,灵灵草` 的顺序轮流使用草种；
    - 日志中会说明已关闭灵性检测，并打印当前轮到的不灵草/灵灵草。

### 草种筛选与预知草精

- **保护草种列表（`--yield-protect-kusa`）**  
  判断依据为页面「XX正在生长中」中的「XX」，与网页下拉框中的选择无关。

- **仅保护草种（不设 `--yield-threshold`）**  
  只指定 `--yield-protect-kusa` 时：每轮生草后检查当前草种；若在保护列表中则保留并进入等待完成；若不在则寻找并点击「除草」后重新生草，如此循环直到出现保护草种或达到 `--yield-max-retry` 次（为 0 时按 10 次）。未找到「除草」按钮时会在当前页面重试寻找（最多 3 次，每次间隔 1 秒），找到并点击后再继续生草；连续 3 次未找到则保留当前草。

- **预知草精 / 生草数 / 沼气倍率 + 保护草种**  
  筛选优先级固定为：**① 草种**（保护列表命中则直接保留）→ **② 连号**（由 `--yield-lianhao-min` 控制）→ **③ 草精数**（`--yield-threshold`，若配置）→ **④ 生草数**（`--yield-grass-min`，若配置）→ **⑤ 沼气倍率**（`--yield-biogas-min`，若配置；页面「沼气×1.n」需**大于**设定值才保留）。任一后续项不达标则除草重试，直至达标或达到 `--yield-max-retry`。

### 游戏地址（默认）

脚本的默认游戏地址为：

`http://110.41.149.62/kusa`

可通过前文的 `--url` 或环境变量 `KUSA_GAME_URL` 覆盖。
